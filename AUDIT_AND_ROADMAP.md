# tusi'm — полный аудит и дорожная карта

Сервис: краудсорсинг фото на мероприятии. Организатор создаёт событие → гости сканируют QR → снимают/загружают фото с лимитом кадров → общий альбом в реальном времени, свайп-голосование, реакции.

---

## 1. Что уже есть

Экраны: лендинг, кабинет организатора (логин по Telegram, дашборд, создание события, кабинет события), гость (QR → камера → загрузка), альбом (realtime + реакции), свайп-голосование, админка.

Готово за прошлые апдейты:
- Безопасность: серверные мутации + RLS-миграция, httpOnly-сессии организатора и админа, гостевые записи только через сервер.
- Реальное создание события с привязкой к организатору, дашборд со статистикой/поиском/фильтром.
- Функции: ZIP-скачивание, гейт гостя по лимиту (`/api/check-guest`), лиды с лендинга.
- Дизайн-система v2: токены, палитра красный+фиолет, единый набор иконок, иллюстрации пустых состояний, анимированный фон, тосты.

Таблицы БД: `events`, `photos`, `organizers`, `reactions`, `leads`.

## 2. Найденные баги и нестыковки (чинить в первую очередь)

- **`/live` не существует**, но ссылка на него есть в админке (📺 Live). Слайдшоу на экран — заявленная фича топ-тарифов, страницы нет.
- **Лимиты тарифа не проверяются на загрузке.** `/api/upload` не смотрит `photo_limit`/`guest_limit` — гость может залить больше, чем по тарифу.
- **Watermark не реализован.** Тарифы обещают «Watermark tusi'm» на бесплатном и «без watermark» на платных — логики нет.
- **Нет удаления события** организатором (только «закрыть»). Накапливается мусор, нельзя убрать тестовое.
- **Обложка события** заявлена в тарифах — не реализована.
- **Оплата — заглушка.** Тарифы показаны, но создание бесплатно для всех; платёжной интеграции нет.

## 3. Чего не хватает «такому сервису» (по категориям)

### A. Продукт / функции
- Слайдшоу `/live` для проектора (живой поток фото на экран зала).
- Обложка события + брендирование страницы гостя (логотип/цвет организатора).
- Лидерборд «фото вечера» по лайкам (итог свайпа), победители.
- Модерация: скрыть/одобрить кадр до показа в общем альбоме.
- Скачать одно фото / поделиться в соцсети-мессенджеры (для гостя).
- Печатный QR-постер (PDF «Сканируй и загружай») на столы.
- Редактирование события (название, длительность, обложка) — сейчас минимально.
- Сжатие фото на клиенте перед загрузкой (экономия трафика/хранилища, скорость).

### B. Монетизация
- Реальная оплата (ЮKassa / Robokassa / CloudPayments / Telegram Payments).
- Логика тарифа: enforce лимитов, watermark на free, разблокировка фич по плану.
- История платежей / счёт организатора.

### C. Прод-готовность / инфраструктура
- PWA: `manifest.json` + иконки + «на главный экран» (гость на телефоне).
- Страницы `not-found` (404) и `error` (500) — брендовые.
- `loading.js` состояния для маршрутов.
- SEO: метаданные по страницам, OG-картинка, `robots.txt`, `sitemap.xml`.
- Фавикон и набор иконок приложения, og-image.
- Защита от абьюза: rate-limit на `/api/upload`, `/api/lead`, размер/тип файла.
- Очистка Storage при удалении события/фото (чтобы не копился мусор).
- Аналитика (privacy-friendly): просмотры, конверсия лендинга, загрузки.

### D. Юридическое (важно для РФ-запуска)
- Политика конфиденциальности, Пользовательское соглашение, согласие на обработку ПДн (152-ФЗ).
- Баннер cookie/согласие, страница «Контакты/Оферта».
- Правила хранения и удаления фото (срок жизни события/данных).

### E. Качество / доступность
- Доступность (WCAG): фокус-стейты, контраст, alt, клавиатура.
- i18n RU/EN (для интернациональных мероприятий) — опционально.
- Уведомления организатору (Telegram) о первых загрузках / заполнении лимита.

## 4. Дорожная карта (этапы)

### Этап 1 — Починки и прод-готовность (фундамент перед запуском)
1. Создать `/live` слайдшоу (или временно убрать ссылку из админки).
2. Enforce лимитов тарифа на `/api/upload` (photo_limit на устройство, guest_limit).
3. Watermark на бесплатном тарифе (наложение логотипа при загрузке/выдаче).
4. Удаление события организатором (+ очистка Storage).
5. Брендовые `not-found`, `error`, `loading`, фавикон/иконки.
6. PWA-манифест + «на главный экран».
7. Сжатие фото на клиенте перед загрузкой.

### Этап 2 — Монетизация
1. Платёжный провайдер (выбрать) + вебхук подтверждения оплаты.
2. Привязка плана к событию после оплаты, разблокировка фич.
3. История платежей в кабинете.

### Этап 3 — Рост и «вау»-функции
1. Слайдшоу-режим довести до премиума (анимации, QR на экране).
2. Обложка события + брендирование страницы гостя.
3. Лидерборд «фото вечера», экран итогов свайпа.
4. Модерация кадров.
5. Шеринг/скачивание отдельного фото гостем.
6. Печатный QR-постер (PDF).

### Этап 4 — Маркетинг и юридическое
1. SEO (метаданные, OG, robots, sitemap), аналитика.
2. Юридические страницы + согласие на ПДн (152-ФЗ).
3. Уведомления организатору, дожимы лидов.
4. i18n RU/EN (если нужно).
